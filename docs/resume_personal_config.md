## Output Format: Always LaTeX (.tex)

Every resume generated by this system MUST be output as a valid LaTeX document.

- Format: `.tex` using the structural template at `docs/templates/resume_template.tex`
- Compiler target: `pdflatex` (letterpaper, 11pt)
- The `.tex` file is the canonical artifact ŌĆö PDF is compiled from it
- Never output DOCX, plain text, or Markdown as the final resume artifact

A Markdown `.md` preview is also generated alongside the `.tex` for quick human
review and light edits. If the user edits the `.md` preview and saves, the system
re-compiles to `.tex`/`.pdf` automatically. This lets you tweak a bullet or a
summary line without touching raw LaTeX.

### Appending to Applications
When a resume is generated for a job application:
1. `.tex` source stored in `applications.latex_content` (TEXT column in DB)
2. Compiled PDF path stored in `applications.tailored_resume_path`
3. Committed to GitHub vault at:
   `resume-vault/applications/{VersionTag}/resume.tex`
   `resume-vault/applications/{VersionTag}/resume.pdf`
4. `resume_usages` table records which vault entry was used and the ATS score at time of use
5. Extension attach flow: when the user is on a job application form, the sidepanel
   surfaces the generated PDF for direct attachment. The content script pre-fills
   the file upload input automatically.

## Resume File Naming (Version Tagging)

### Two Separate Naming Concerns

| Context | Format | Example |
|---------|--------|---------|
| **PDF seen by recruiter / hiring manager** | `{FirstName}.pdf` only | `Narendranath.pdf` |
| **Git tag / internal versioning** | `{FirstName}_{CompanyShortName}_{RoleAbbrev}` | `Narendranath_Google_DE` |
| **Git tag when same role applied multiple times** | `{FirstName}_{CompanyShortName}_{RoleAbbrev}_{JobID}` | `Narendranath_Google_DE_JOB123` |

**Why `{FirstName}.pdf` only for recruiters:**
A file named `Narendranath_Google_DE_v3.pdf` tells the recruiter you have a version
specifically crafted for them ŌĆö which breaks the illusion of a strong natural fit.
`Narendranath.pdf` is clean, confident, and professional.

**Why JobID in git tags:**
When you apply to Google DE twice (different job postings), the git tag uniquely
identifies each submission without polluting the `versions/` flat directory.
The `versions/` directory still holds one canonical file ŌĆö the most recently accepted
version. Git history shows all iterations.

## Resume Integrity Rules (Critical)

### Facts-Only Policy
The system MUST only use information confirmed in the user's stored resume history.
- **Never fabricate metrics, percentages, or outcomes** not in the original resume
- **Never add technologies or tools** the user has not used
- Every bullet must be traceable to the user's actual work history

### Coherent Story Principle
A resume is a narrative. The skills and experience must add up as one consistent story.

**Cloud Platform Alignment Rule:**
Do NOT list all cloud platforms (Azure, AWS, GCP) as primary experience in one resume.
A candidate who is "expert in Azure, AWS, and GCP equally" in the same resume is
unconvincing to experienced hiring managers. Instead:
- Identify the target company's primary cloud platform (research from JD + company knowledge)
- Lead with the platform they use
- Mention others in supporting context only (e.g., "also deployed on AWS")
- Example: For an Azure-heavy role, lead with Azure/Fabric experience; mention AWS/GCP briefly

**Skill Selection Rule:**
Only include skills the user can credibly defend in a 30-minute technical interview.
The system must NOT add a skill just because it appears in the JD and would increase
the ATS score. A false skill that gets tested in an interview is far worse than a missing
one on the resume.

**The Right ATS Approach:**
Increase ATS score by reframing genuine experience using the JD's vocabulary ŌĆö
NOT by adding skills the user doesn't have. A 72% ATS score built on real skills
is better than an 88% score built on inflated claims.

## Single Page Enforcement

The system enforces single-page output:
- Estimate content height after generating all sections
- If content exceeds ~55 lines of body text, trim in this order:
  1. Reduce oldest experience role to 2-3 bullets
  2. Drop least-relevant project
  3. Condense certifications line
  4. Trim summary to 2-3 sentences
- Never trim: contact header, education, current/most recent role, skills section

## LLM Generation Cascade

Try providers in this order:
1. **Anthropic** `claude-sonnet-4-6` ŌĆö preferred; best LaTeX structure adherence
2. **OpenAI** `gpt-4o` ŌĆö fallback paid option
3. **Kimi** `moonshot-v1-32k` ŌĆö for very long JDs (>8K tokens)
4. **Ollama local** `llama3.1:8b` ŌĆö offline/on-machine; no API cost
5. **Keyword fallback** ŌĆö rule-based only, no LLM; returns partially filled template

### Local Model Recommendation
- Generation: `llama3.1:8b` via Ollama
- Embeddings: `nomic-embed-text` via Ollama (768-dim, excellent for doc retrieval)

### Why This Structure

**`versions/` (flat, named files):**
- Always contains the *current best* resume for each company/role combination
- Matches the `Narendranath_Google_DE` naming convention exactly ŌĆö no version numbers visible
- Clean to browse on GitHub; one glance shows every company you're targeting
- Overwritten in-place when you regenerate for the same company+role ŌĆö old state preserved by git history

**`applications/` (dated subdirectories):**
- Every submission attempt is preserved permanently with its exact date
- Multiple attempts at the same company show as separate dated folders
- `metadata.json` captures the JD hash, ATS score at time of submission, LLM strategy used, and changes summary

**Git Tags for named versions:**
- Each time `versions/Narendranath_Google_DE` is created or updated, the system creates/moves a Git tag with the same name
- GitHub surfaces tags at `github.com/{user}/resume-vault/tags` ŌĆö a clean list of every resume version you've ever targeted
- Clicking a tag on GitHub shows the exact commit diff: what changed between this version and the previous one
- You can check out any tag to restore the exact resume state: `git checkout Narendranath_Google_DE`

### How Versions Are Surfaced in the Extension

1. **Resume list**: Extension calls GitHub API to list `versions/` directory ŌåÆ shows all current resumes as cards (displayed by git tag name, e.g. `Narendranath_Google_DE`)
2. **Company history**: When on a career page for Google, extension filters `applications/` for `Narendranath_Google_*` subdirs ŌåÆ shows every past submission with date, ATS score, and outcome
3. **Same-role multi-apply**: If `Narendranath_Google_DE` already exists and user reapplies, the new submission gets tag `Narendranath_Google_DE_{JobID}` ŌĆö both appear in the company history view
4. **Diff view**: "Compare Versions" opens a GitHub diff URL between two tagged commits
5. **Tag browse**: "View all on GitHub" button opens `github.com/{user}/resume-vault/tags`
6. **Download for recruiter**: Always downloads and renames to `Narendranath.pdf` before attaching ŌĆö the internal tag name never leaks to the recruiter

### Offline-First: Markdown Edits ŌåÆ Git Sync

When the user edits a `.md` resume preview file in the sidepanel or any editor:

**Online:** Changes are committed to GitHub immediately via the GitHub API.
Git commit message: `update: {VersionTag} ŌĆö manual edit {timestamp}`

**Offline:** Changes are staged to a local sync queue stored in:
- **Extension context**: `IndexedDB` ŌåÆ `pending_commits` store, keyed by `{VersionTag}_{timestamp}`
- **Desktop/backend context**: `SQLite` local file at `~/.autoapply/offline_queue.db`

On next network availability (`navigator.onLine` event or backend health check):
1. Drain the `pending_commits` queue in chronological order
2. For each entry: fetch current file SHA from GitHub, apply the edit, commit
3. If a conflict is detected (GitHub has a newer version): present a merge diff in the sidepanel ŌĆö user picks which version wins
4. Mark entry as `synced` in local queue

**Implementation note:** Uses `isomorphic-git` (pure JS, works in browser + Node) for
local staging, with the GitHub REST API as the remote. No git binary required.

## Application Attachment Flow (Extension)

When the browser extension detects an application form:
1. Content script identifies file upload fields (resume upload `<input type="file">` elements)
2. Sidepanel shows the best-matched resume from `versions/` for this company/role
3. User reviews ATS score and skills gap before submitting
4. User clicks **"Attach Resume"** in the sidepanel
5. Extension downloads the PDF from `versions/Narendranath_{Company}_{Role}.pdf` via GitHub API
   (falls back to local IndexedDB cache if GitHub is unavailable or slow)
6. Extension programmatically sets the file input's value with the PDF blob
7. The application record in the DB is updated with the resume version tag and ATS score used

### Local Cache Strategy
The extension maintains an IndexedDB cache keyed by `{VersionTag}` ŌåÆ `{pdf_blob, tex_content, metadata, cached_at}`.
Cache TTL: 24 hours. On cache miss, fetch from GitHub. On GitHub error, serve stale cache with a warning.

#### Answer Generation Flow
1. Extract question text + detect category
2. Query `application_answers` for same `question_hash` at same company ŌåÆ surface as **"Previously used"** draft
3. If not found ŌåÆ call LLM: question + company research + user resume facts ŌåÆ **3 draft answers**
4. Each draft: Ōēż 250 words, Ōēł 180-220 words ideal (readable in 6-7 seconds at 130 wpm)
5. User selects a draft OR types **"enter default"** ŌåÆ system auto-selects highest-scored draft
6. Accepted answer saved to `application_answers` with `application_id`, `company_name`, `question_hash`

**Every generated draft must follow these non-negotiable rules:**
- Ground every answer in the user's real, stored work history ŌĆö no fabricated examples
- Align tone and language to the company's known culture (researched from JD + company knowledge base)
- End with a forward-looking statement tying your background to their mission
- Use active voice, first-person implied (no "I am a person who...")

### Answer Frameworks by Category (LLM Grounding)

All answers must be grounded in the user's **real, stored work history**. No fabrication.

#### `why_company` ŌĆö Why do you want to work here?
**Framework: Research ŌåÆ Connect ŌåÆ Contribute**
```
1. Name one specific, researched thing about the company (product, initiative, mission, recent milestone)
2. Connect it to a real experience or value from your history
3. State what you will contribute ŌĆö be specific to their domain
```
> "I've closely followed {Company}'s investment in {specific area}. In my work at {past employer},
> I built {relevant thing} that achieved {metric}. I want to bring that same approach to help
> {Company} {specific goal from JD or company news}."

Never open with "I've always been passionate about..." ŌĆö too generic.

#### `why_hire` ŌĆö Why should we hire you?
**Framework: Match ŌåÆ Prove ŌåÆ Differentiate**
```
1. Pick the top 2 MQs from the JD and match them to your experience with metrics
2. Prove with one specific outcome (use XYZ formula)
3. Add one differentiator not obvious from the resume (cross-functional skills, domain combo, etc.)
```
> "The role calls for {skill A} and {skill B}. At {employer}, I {did X} achieving {metric Y}.
> What sets me apart is {unique angle} ŌĆö I bring both {technical depth} and {business context}
> that I've found is rare in this space."

#### `about_yourself` ŌĆö Tell me about yourself
**Framework: Present ŌåÆ Past ŌåÆ Future (Ōēż 90 seconds spoken, Ōēż 200 words written)**
```
1. Current role + 1 standout achievement (10 seconds)
2. How you got here ŌĆö 2-3 pivotal steps that tell a coherent story (20 seconds)
3. Why this role is the logical next step (10 seconds)
```
Do NOT recite the resume chronologically. Tell a story with a thesis.

#### `strength` ŌĆö Greatest strength
**Framework: Name it ŌåÆ Prove it with STAR ŌåÆ Connect to this role**
```
1. Name the strength clearly and confidently (not "I guess I'm pretty good at...")
2. One specific STAR example with a measurable result
3. Explain how it directly applies to what they need
```
Be specific: "translating ambiguous business problems into data models" beats "I'm analytical."

#### `weakness` ŌĆö Greatest weakness
**Framework: Real weakness ŌåÆ Active improvement ŌåÆ Evidence of progress**
```
1. Name a genuine, non-disqualifying weakness (not "I work too hard")
2. Describe concrete steps you've taken to improve it
3. Show measurable progress or a system you've built around it
```
> "Early in my career, I underestimated the time required for stakeholder alignment on data
> projects. I've since built a habit of scheduling a 30-min kickoff for every new pipeline to
> align on definitions upfront ŌĆö this cut revision cycles by about half."

Avoid clich├®s: perfectionism, caring too much, working too hard.

#### `challenge` ŌĆö Describe a challenge / failure
**Framework: STAR + Lesson**
```
Situation: brief context (1-2 sentences)
Task: your specific responsibility
Action: concrete steps YOU took (use "I", not "we")
Result: measurable outcome
Lesson: what you'd do differently or what you built from it
```
Pick a challenge that shows resilience and self-awareness ŌĆö not a trivial inconvenience.
The lesson is what separates a good answer from a great one.

#### `leadership` ŌĆö Leadership experience
**Framework: Context ŌåÆ Team size + scope ŌåÆ Your specific actions ŌåÆ Outcome**
```
Always include: team size, timeline, and a measurable result.
```
> "I led a team of 4 engineers over 6 months to migrate our legacy ETL to a streaming
> architecture. I ran weekly syncs, unblocked two critical dependencies with platform teams,
> and we delivered 3 weeks ahead of schedule ŌĆö cutting pipeline latency by 70%."

#### `conflict` ŌĆö Conflict with a colleague
**Framework: Situation ŌåÆ Your action (not blame) ŌåÆ Resolution ŌåÆ Relationship preserved**
```
1. Describe the disagreement neutrally ŌĆö no villain
2. What YOU did to address it (active listening, proposed compromise, escalated appropriately)
3. How it resolved and what you learned about collaboration
```
Never blame the other party. Focus on process and communication.

#### `motivation` ŌĆö What motivates you?
**Framework: Intrinsic + Role-specific + Company-connected**
```
1. Name an intrinsic motivator (solving hard problems, seeing data drive decisions, etc.)
2. Connect to a real moment from your history where that motivation led to an outcome
3. Tie it to what this role/company offers
```

#### `five_years` ŌĆö Where do you see yourself in 5 years?
**Framework: Growth ŌåÆ Contribution ŌåÆ Flexibility**
```
1. Describe the type of work you want to master ŌĆö not a specific title
2. Show alignment with where the company is headed
3. Express genuine flexibility ("within a growing company like this, that path could take shape in several ways")
```
Don't say "your job" or "in your company" ŌĆö sounds rehearsed. Be aspirational but not rigid.

#### `impact` ŌĆö Most proud of / greatest achievement
**Framework: XYZ formula + Why it matters to you personally**
```
Accomplished [X] as measured by [Y], by doing [Z].
Then: "What made this meaningful was..."
```
The personal reflection is what distinguishes this from a resume bullet.

#### `fit` ŌĆö Culture fit / values alignment
**Framework: Research-specific ŌåÆ Personal value ŌåÆ Evidence from history**
```
1. Name a specific company value (from their website/JD ŌĆö not generic "innovation")
2. Show you hold that value with a real example
3. Connect to what you'll bring to the culture
```

### Recruiter Callback Reference

When a recruiter calls back, the sidepanel surfaces for that company + application:

- **Resume version used**: git tag (e.g. `Narendranath_Google_DE_JOB123`), ATS score at submission
